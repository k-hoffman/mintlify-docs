A checklist template is a structured list of fields you want Qumis to extract from insurance documents. You write it as a simple text outline, upload your documents, and Qumis reads each one to fill in the values — producing a clean, side-by-side coverage table you can use in proposals, renewals, and client presentations. This page walks you through how to write templates, handle repeating items like locations and vehicles, and use line-of-business examples to get started quickly.

---

## What checklist templates do

A template tells the [Coverage Table](/platform/populate) exactly what data to pull from your policies. Instead of Qumis deciding what to extract, you define the fields — and Qumis fills them in for every document you upload.

With a custom checklist, you can:

- Define the exact coverage fields, limits, and terms you need extracted
- Handle repeating items like locations, buildings, vehicles, and drivers automatically
- Compare multiple documents side by side with a consistent set of fields
- Produce output that matches your firm's internal formats and client deliverables

Templates are entered in the **Additional Notes** field when creating a Coverage Table report.

---

## Before you start

- Decide which line of business you're working with (e.g., Commercial Property, Auto/Fleet, Workers' Comp, D&O)
- Have a sense of the specific fields you want extracted — limits, deductibles, exclusions, sublimits, etc.
- Know whether your documents have repeating items (multiple locations, buildings, vehicles) that need to expand automatically
- Have at least one policy document ready to test your template against

---

## How templates work

### One field per line

Each line in your template represents a single data point you want extracted. Qumis reads the document and returns a value for each field.

### Indentation shows grouping

Indent child fields under their parent with spaces. This tells Qumis that the indented fields belong to the parent group.

### Flat fields get one value

Fields like "Policy Number" or "Effective Date" produce a single extracted value per document.

### List fields expand automatically

Fields like "Locations" or "Vehicles" repeat based on how many the system finds in each document. You define the structure once — Qumis discovers the actual count from the documents.

## Write your first template

<Steps>
  <Step title="Start with policy-level fields">
    Begin with the flat fields that appear once per policy — identifiers, dates, and overall limits. These go at the top with no indentation.

    ```plaintext  theme={null}
    Named Insureds
    ERISA Name
    DBA
    Mailing Address
    Policy Number
    Effective Date
    Expiration Date
    Property Premium
    ```

    Each field produces one row in the output, with a value extracted from each document.
  </Step>

  <Step title="Add repeating groups for multi-item data">
    When a policy has multiple locations, buildings, vehicles, or other repeating items, indent the detail fields under the group name.

    ```plaintext  theme={null}
    Locations
      Premises Information
      Premises Description
      Buildings
        Address
        Limit
        Cause of Loss
        Deductible
        Coinsurance
        Valuation
    ```

    Qumis reads each document and figures out how many locations exist, then how many buildings are at each location. It creates a row for every combination automatically.
  </Step>

  <Step title="Test your template">
    Enter the template in the **Additional Notes** field when creating a Coverage Table report. Upload a document and generate the report. Review the output to see if your fields are being extracted as expected, then adjust as needed.
  </Step>

  <Step title="Save your template to the Prompt Library">
    Once your template works well, save it to your [Prompt Library](/platform/prompt-library) so you and your team can reuse it. Give it a descriptive name like "Commercial Property Checklist" or "Auto Fleet Extraction Template."
  </Step>
</Steps>

## How repeating groups expand

To understand how nesting works, consider a Commercial Property template with Locations and Buildings. If a policy has 2 locations and Location 1 has 2 buildings while Location 2 has 1 building, Qumis creates rows for every combination:

- Location 1 → Premises Information
- Location 1 → Premises Description
- Location 1 → Building 1 → Address, Limit, Deductible...
- Location 1 → Building 2 → Address, Limit, Deductible...
- Location 2 → Premises Information
- Location 2 → Premises Description
- Location 2 → Building 1 → Address, Limit, Deductible...

<Tip>
  You do not need to know the count ahead of time. Just define the structure once and the system discovers the actual number from each document.
</Tip>

## Recognized repeating groups

Qumis automatically recognizes these field names as repeating groups when they have indented fields beneath them:

| Recognized name | Typical use case |
| --------------- | ---------------- |
| Locations       | Commercial Property, General Liability |
| Buildings       | Property (nested under Locations) |
| Vehicles        | Auto, Fleet, Commercial Auto |
| Drivers         | Auto, Fleet |
| Items           | Inland Marine, Scheduled items |
| Equipment       | Contractor's Equipment, EDP |
| Premises        | General Liability, BOP |
| Properties      | Property schedules |
| Vessels         | Marine, Hull |
| Aircraft        | Aviation |
| Trailers        | Trucking, Motor Carrier |
| Units           | Manufactured Housing, Mobile Equipment |
| Schedules       | Scheduled coverages or endorsements |

<Warning>
  The name must match exactly. "Buildings" is recognized as a repeating group, but "Building Information" or "Building Details" would not be — those are treated as regular single-value fields.
</Warning>

---

## The \[] marker for custom repeating groups

If you need a field to repeat that isn't in the recognized list above, add `[]` after the name:

```plaintext  theme={null}
Named Insureds[]
  Name
  Role
  Address
```

This tells Qumis explicitly: "there can be more than one of these." The `[]` is removed from the output — you'll see "Named Insureds" in your results, not "Named Insureds\[]".

You can also use `[]` on recognized fields for extra clarity:

```plaintext  theme={null}
Locations[]
  Premises Description
  Buildings[]
    Address
    Limit
    Valuation
```

Both approaches work. The `[]` marker simply removes any ambiguity.

---

## Fields that do NOT repeat

These common fields are always treated as single values, even though they sound plural:

- Named Insureds (unless you add `[]`)
- Coverages
- Exclusions
- Endorsements
- Each Occurrence
- Per Claim Retention
- Aggregate Limit
- Premium

If you need any of these to repeat, add the `[]` marker.

## Best practices

- **Start with policy-level fields at the top** — policy number, dates, and overall limits should come first with no indentation.
- **Group repeating items below the flat fields** — indent detail fields under their parent group name.
- **Use consistent indentation** — 2 or 4 spaces, just be consistent throughout the template.
- **Create separate templates per line of business** — a Commercial Property checklist and an Auto checklist work better than one massive combined template.
- **Use the `[]` marker when in doubt** — if you're not sure whether a field will be recognized as a repeating group, adding `[]` ensures it expands correctly.
- **Nesting works to any depth** — Locations > Buildings > Equipment is fully supported. Define as many levels as your documents require.
- **Test on a sample document first** — run your template against one policy before using it on a batch to confirm the fields extract as expected.
- **Save working templates to the Prompt Library** — once a template produces good results, save it so your team can reuse it consistently.
- **When comparing policies**, upload multiple documents and use the same template. Qumis extracts values from each document side by side.

## Troubleshooting

<AccordionGroup>
  <Accordion title="A field is returning blank values">
    The field name may not match how the document describes that data. Try using the exact terminology from the policy (e.g., "Self-Insured Retention" instead of "SIR"). If the data isn't in the document, Qumis will leave the field blank.
  </Accordion>

  <Accordion title="My repeating group isn't expanding">
    Make sure the group name matches one of the recognized repeating group names exactly, or add `[]` after the name. "Buildings" is recognized, but "Building Details" is not. Also verify that the detail fields are indented under the group name.
  </Accordion>

  <Accordion title="I'm getting too many rows in the output">
    This usually means a field is being treated as a repeating group when you intended it as a flat field. Check that only fields you want to repeat have indented children beneath them.
  </Accordion>

  <Accordion title="Fields are appearing in the wrong order">
    Qumis extracts fields in the order they appear in your template. Rearrange the lines in your template to match the order you want in the output.
  </Accordion>

  <Accordion title="I need a field to repeat but it's in the 'does not repeat' list">
    Add the `[]` marker after the field name. For example, `Named Insureds[]` will force the field to expand for each named insured found in the document.
  </Accordion>

  <Accordion title="My template works for one policy but not another">
    Different carriers use different terminology and document structures. You may need to adjust field names to match the language in each carrier's policy, or create carrier-specific template variants.
  </Accordion>
</AccordionGroup>

## Why use custom checklists?

Qumis can extract coverage data automatically, but custom checklists give you control over exactly what gets pulled and how it's organized. When your firm has a standard format for coverage summaries, renewal comparisons, or compliance verifications, a checklist template ensures every report matches that format — no manual reformatting required. Teams that standardize their templates produce more consistent output, reduce review time, and build a library of reusable extraction patterns that improve with every use.
