## How it works

Documents are ingested, validated, converted to PDF if needed, processed through OCR and data extraction, and returned as structured JSON. Every document passes through the same processing stages:

1. **File validation** — Qumis confirms the file type is supported and checks for pre-existing records to avoid duplicate processing.
2. **OCR processing** — Scanned documents and images are converted to machine-readable text.
3. **Data extraction** — Qumis identifies and extracts structured policy information (limits, deductibles, endorsements, exclusions, etc.).
4. **Output generation** — Extracted data is formatted as structured JSON for downstream consumption.
5. **Exception handling** — Any documents that fail validation or extraction are flagged with detailed error information for manual review.

## API process

The API process lets your organization send documents to Qumis individually via REST API and receive results through webhook notifications. This method is ideal for real-time workflows, on-demand extraction, and event-driven architectures.

<Steps>
  <Step title="Upload to DMS">
    An employee uploads policy documents to your Document Management System (DMS).
  </Step>
  <Step title="Backend picks up files">
    Your backend job detects new documents and converts them to PDF format if needed.
  </Step>
  <Step title="API request">
    Your backend sends a POST request to the Qumis API to generate a report, including the PDF document.
  </Step>
  <Step title="Report record created">
    Qumis creates a report record and begins processing. The API returns a report ID for tracking.
  </Step>
  <Step title="Validation & extraction">
    Qumis validates the document, checks for pre-existing records, runs OCR (if needed), and extracts structured policy data.
  </Step>
  <Step title="Webhook notification">
    When processing completes, Qumis sends a webhook notification to your configured endpoint.
  </Step>
  <Step title="Retrieve results">
    Your backend calls the Qumis API (POST) to retrieve the extraction results as structured JSON. If an error occurred, the response includes detailed error information.
  </Step>
  <Step title="Load to warehouse">
    Your backend transforms the JSON data and loads it into your data warehouse.
  </Step>
</Steps>

### API integration requirements

| Component | Details |
|---|---|
| **API access** | Qumis provides API credentials and endpoint documentation |
| **File format** | PDF (your backend handles conversion from native formats) |
| **Webhook endpoint** | Your backend exposes an HTTPS endpoint to receive completion notifications |
| **Error handling** | Your team monitors API error responses and handles remediation |

---

## Exception handling

When documents fail validation or extraction, Qumis flags them with detailed error information in the API error response. Your team reviews flagged documents and remedies them before reprocessing.

Common exception reasons include unsupported file types, duplicate documents, OCR failures from poor scan quality, and extraction confidence falling below threshold.

## Configure your extraction template

Define your own data extraction template to control which fields Qumis pulls from each document. Fill out the spreadsheet template below and submit it to your Qumis representative to configure your custom extraction list.

<Steps>
  <Step title="Download template">
    Download the template spreadsheet in CSV or Excel format.
  </Step>
  <Step title="Define fields">
    Fill in one row per field you want to extract.
  </Step>
  <Step title="Submit to Qumis">
    Send the completed template to your Qumis representative.
  </Step>
</Steps>

## Template columns

| Column | Required | Description |
|---|---|---|
| **key** | Yes | Unique identifier for this field (lowercase, underscores, no spaces) |
| **label** | Yes | Display name shown in reports |
| **instructions** | Yes | Detailed instructions for the AI on how to find and extract this value |
| **example** | Yes | Example of the expected output format (see Value Types below) |
| **mapping** | Yes | Value type: `1` (single), `M` (multiple), or `C` (complex) |
| **category** | No | Optional grouping for organizing fields in reports |

---

## Value types (mapping)

### Type `1` — Single value

Use for fields that have exactly one value — policy numbers, dates, names, single amounts (premium, deductible), or yes/no fields.

**Example format:**

```json
{"field_key": "value"}
```

| Field | Example |
|---|---|
| Policy Number | `{"policy_number": "POL-2024-001"}` |
| Effective Date | `{"effective_date": "2024-01-15"}` |
| Premium Amount | `{"premium": 125000}` |
| Insured Name | `{"insured_name": "Acme Construction Inc."}` |

### Type `M` — Multiple values

Use for fields that return a list of simple values — lists of perils, exclusions, locations, multiple selections, or tags.

**Example format:**

```json
["value1", "value2", "value3"]
```

| Field | Example |
|---|---|
| Covered Perils | `["Fire", "Flood", "Windstorm", "Theft"]` |
| Exclusions | `["War", "Nuclear", "Pollution"]` |
| Named Insureds | `["ABC Corp", "XYZ Holdings", "Smith LLC"]` |
| Locations | `["Toronto, ON", "Vancouver, BC"]` |

### Type `C` — Complex object

Use for fields that return structured data with multiple properties — tables or grids, items with multiple attributes, or nested information.

**Example format:**

```json
[
  {"property1": "value1", "property2": "value2"},
  {"property1": "value3", "property2": "value4"}
]
```

| Field | Example |
|---|---|
| Coverage Limits | `[{"coverage": "Property", "limit": 5000000}, {"coverage": "Liability", "limit": 2000000}]` |
| Sublimits | `[{"peril": "Flood", "amount": 500000}, {"peril": "Earthquake", "amount": 250000}]` |
| Named Insureds with Role | `[{"name": "ABC Corp", "role": "Primary"}, {"name": "XYZ Inc", "role": "Additional"}]` |

## Writing good instructions

The **instructions** field tells the AI how to find and extract the value. Good instructions lead to accurate extractions.

### Best practices

- **Be specific** — Describe exactly what to look for.
- **List alternative names** — Include different terms that might be used.
- **Specify format** — Tell the AI how to format the output.
- **Provide context** — Explain where to look in the document.

### Instruction template

<Note>
  Extract [what to find]. [Where to look]. [Format instructions]. [Alternative terms].
</Note>

### Examples

| Field | Good Instruction |
|---|---|
| **Policy Number** | "Extract the unique policy identifier. Look in the declarations page header or policy information section. This may be labeled as 'Policy Number', 'Policy No.', 'Certificate Number', or 'Policy ID'." |
| **Effective Date** | "Extract the policy start date in ISO format (YYYY-MM-DD). Look for 'Effective Date', 'Policy Period From', 'Inception Date', or 'Coverage Begins'. If only month/year is shown, use the 1st of the month." |
| **Deductible** | "Extract the per-occurrence deductible amount as a number without currency symbols. Look in the deductible schedule or declarations. May be labeled as 'Deductible', 'Retention', or 'Self-Insured Retention (SIR)'." |
| **Covered Perils** | "Extract all perils explicitly covered by the policy as an array. Look in the 'Covered Causes of Loss' or 'Insured Perils' section. Include both named perils and any 'all-risk' designation." |

## Common mistakes to avoid

<Warning>
  Review these common pitfalls to ensure your template is configured correctly.
</Warning>

| Mistake | Problem | Solution |
|---|---|---|
| Using spaces in key | `policy number` | Use underscores: `policy_number` |
| Vague instructions | "Get the limit" | Be specific: "Extract the per-occurrence limit of liability from the declarations page" |
| Wrong mapping type | Using `1` for a list | Use `M` for lists, `C` for structured data |
| Missing example | No example provided | Always provide a realistic example |
| Inconsistent JSON | `{'key': 'value'}` | Use double quotes: `{"key": "value"}` |

---

## Validation checklist

Before submitting your template, verify:

- All `key` values are unique
- All `key` values use lowercase and underscores only
- All required columns are filled (key, label, instructions, example, mapping)
- Mapping values are only `1`, `M`, or `C`
- Examples use valid JSON format
- Instructions are detailed enough for accurate extraction
